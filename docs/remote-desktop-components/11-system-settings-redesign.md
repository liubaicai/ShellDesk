# 系统设置组件重设计文档

> 当前状态：已接入远程桌面（appKey: `settings`），实现入口为 `src/components/remote-desktop/RemoteSettings.tsx`。本文保留重设计背景，维护时以当前实现、`RemoteDesktopShell.tsx` 注册表和 `_example.md` 清单为准。

## 定位

系统设置是远程主机配置入口，但不应成为所有运维功能的杂物箱。重新设计时它应聚焦系统级配置，和服务、包管理、网络诊断、Hosts/DNS 等专门组件建立清晰分工。

## 重新设计目标

- 让系统基础配置按主题组织，减少巨型页面堆叠。
- 将查看、编辑、应用、回滚提示分清。
- 对权限不足、系统差异和危险配置提供明确反馈。
- 把更专门的工具拆出去，让设置只保留配置入口。

## 功能架构

### 系统概览

- 主机名、系统版本、时区、内核、磁盘摘要。
- 当前用户与权限提示。

### 配置分区

- 主机名与基础标识。
- 网络接口和静态配置。
- DNS 与路由入口。
- 软件源/镜像源。
- 系统更新入口。
- Hosts 入口可跳专门工具。

### 变更流程

- 加载当前配置。
- 编辑草稿。
- 预览将执行的命令或文件 diff。
- 应用并显示结果。

## 交互设计

- 左侧为设置导航分组。
- 右侧为当前设置页，保持表单密度。
- 页面顶部固定显示系统类型和权限状态。
- 危险操作，如修改网络、主机名、更新系统，都要有确认和失败回显。

## 数据与状态

```ts
interface RemoteSettingsSectionState<T> {
  loaded: boolean;
  loading: boolean;
  current?: T;
  draft?: T;
  error?: string;
  success?: string;
}
```

## 能力与集成设计

- 设置页大量能力依赖 `runCommand` 和文件读写，命令构建应按系统类型模块化。
- 包安装升级应跳包管理器中心，不在设置页复制整套体验。
- Hosts/DNS、网络诊断、防火墙可在设置中保留入口。
- 修改网络配置必须特别考虑 SSH 断连风险。

## 开发计划

1. 定义设置边界和分区导航。
2. 抽系统类型 provider，隔离 Linux/Windows 命令。
3. 给每个设置页建立 current/draft/apply 状态。
4. 为网络、主机名、镜像源增加预览与确认。
5. 将专门能力改为跳转入口。
6. 增加权限不足和 SSH 断连风险提示。
7. 补跨发行版配置读取测试。

## 验收标准

- 设置导航清楚，不依赖长页面滚动找功能。
- 每个可修改设置都有草稿和应用反馈。
- 网络类危险修改有强提示。
- 无权限或不支持的系统有明确降级。
- 与专门组件边界不重复冲突。

## 设计取舍

- 设置不是运维功能全集。
- 越接近系统级破坏性配置，越要优先预览和确认。
- 专门问题交给专门组件，设置提供系统入口和基础配置。
