# 系统监视器组件重设计文档

## 定位

系统监视器用于回答“这台远程主机现在健康吗”。重新设计时它应提供稳定的摘要、趋势和诊断跳转，而不只是展示几个瞬时数字。

## 重新设计目标

- 将 CPU、内存、磁盘、网络、负载和系统信息组织成可扫描面板。
- 为异常指标提供跳转入口：进程、磁盘分析、端口、日志。
- 区分瞬时指标、采样趋势和系统静态信息。
- 在不同系统上优雅降级。

## 功能架构

### 总览

- 主机名、系统、运行时间、负载、CPU、内存、磁盘、网络。
- 风险摘要：磁盘高水位、内存压力、负载异常。

### 趋势

- 最近几分钟 CPU、内存、网络吞吐采样。
- 当前窗口内缓存即可，不要求历史数据库。
- 采样间隔可控，窗口失焦时降频。

### 诊断入口

- CPU/内存异常跳进进程管理器。
- 磁盘异常跳进磁盘空间分析器。
- 网络摘要跳进端口或网络诊断。
- 系统信息复制为排障摘要。

## 交互设计

- 顶部为主机和健康摘要。
- 中部为紧凑指标图和资源表。
- 底部为静态系统信息和跳转操作。
- 不把每个指标装进过度装饰卡片，优先比较与扫描。

## 数据与状态

```ts
interface MonitorSample {
  timestamp: string;
  cpuPercent?: number;
  memoryPercent?: number;
  netRxBytes?: number;
  netTxBytes?: number;
}

interface MonitorWarning {
  key: string;
  level: 'info' | 'warning' | 'danger';
  message: string;
}
```

## 能力与集成设计

- 继续复用系统状态和 metrics IPC。
- 如果需要更多磁盘/网络细节，保持监控摘要轻量，把深入分析交给专门组件。
- 采样应有清理机制，窗口关闭即停止 polling。
- 支持复制系统摘要给日志或便签。

## 开发计划

1. 重整指标摘要和静态信息布局。
2. 增加采样缓冲和轻量趋势图。
3. 增加 warning 规则。
4. 增加向进程、磁盘、网络组件跳转。
5. 补 Windows/Linux 可用字段差异。
6. 做 polling 生命周期和性能检查。
7. 深色/浅色主题校验。

## 验收标准

- 监控窗口能持续刷新且关闭后停止请求。
- 指标缺失时 UI 有降级而不崩溃。
- 健康摘要和趋势含义清楚。
- 异常指标能进入对应工具。
- 系统摘要可复制。

## 设计取舍

- 不在监控中堆完整进程表、日志表和磁盘扫描。
- 首版趋势只覆盖当前会话。
- 监视器应是入口和摘要，不是所有运维功能的容器。
